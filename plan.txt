SmartDNSSort IP 优选逻辑整改计划
一、 目标 (Goal)
解决缓存过期后首位 IP 失效的逻辑漏洞，通过 “全局 IP 池主动巡检” 代替 “域名被动触发测速”，实现 CDN 级别的实时 IP 优选与复用，显著降低 ICMP 频率。

二、 第一阶段：建立全局 IP 资源管理器 (IP Infrastructure)
新建 IPPool 模块：在 ping 包中实现，维护一个 IP -> {代表性域名, 引用计数, 访问热度} 的结构。
增量更新机制：联动 Cache.SetSorted，在域名 IP 发生变化时，实时增减 IP 池的引用计数。
SNI 绑定映射：自动选取该 IP 下引用数最高或访问最频发的域名作为测速时的 代表性域名。
三、 第二阶段：实现主动巡检调度器 (Proactive Monitor) ✅ 已完成
优先级调度算法：计算重量权重 $W = (RefCount \times A + AccessHeat \times B)$，权重越高测速频率越高。
三级分步刷新：
T0 核心池：2-3 分钟刷新（最高实时性）。
T1 活跃池：10-15 分钟刷新。
T2 淘汰池：长效巡检或不通即剔除。
背景协程落地：在

dnsserver/tasks.go
 中启动 GlobalIPMonitor。

实现文件：
- ping/ip_monitor.go: IP 主动巡检调度器核心实现
- dnsserver/tasks.go: 添加 startIPMonitorRoutine 方法
- dnsserver/server.go: 添加 ipMonitor 字段
- dnsserver/server_init.go: 初始化 IP 监控器
- dnsserver/server_lifecycle.go: 启动和停止 IP 监控器
四、 第三阶段：SNI 感知与混合探测优化 (Smart Probing)
SNI 注入：改造 Pinger.smartPing，在 TLS 握手与 HTTP Fallback 时自动从 IPPool 提取 Host 和 ServerName。
测速逻辑切换：将“用户请求触发测速”降级为“兜底方案”，首选路径改为直接读取 IPPool 维护好的 RTT 数据。
五、 第四阶段：配置与监控 (Configs & Metrics) ✅ 已完成
新增配置参数：
- ip_monitor.enabled: 是否启用 IP 监控器
- ip_monitor.max_size: IP 池最大容量 (默认 1000)
- ip_monitor.t0_refresh_interval: T0 核心池刷新间隔 (默认 120 秒)
- ip_monitor.t1_refresh_interval: T1 活跃池刷新间隔 (默认 900 秒)
- ip_monitor.t2_refresh_interval: T2 淘汰池刷新间隔 (默认 3600 秒)
- ip_monitor.ref_count_weight: 引用计数权重 (默认 1.0)
- ip_monitor.access_heat_weight: 访问热度权重 (默认 0.5)
- ip_monitor.max_refresh_per_cycle: 每次刷新最大 IP 数 (默认 50)
- ip_monitor.refresh_concurrency: 并发测速数量 (默认 10)

API/看板展示：
- /api/ip-pool/status: 获取 IP 池状态和监控统计
- /api/ip-pool/top: 获取 Top IP 列表及其 RTT、引用计数、代表性域名

  - Top IP 列表（IP、RepDomain、RefCount、AccessHeat、RTT、LastAccess）
