# 技术补充总结

## 概述

通过对比工作0（resolver 现状分析）和工作1-8（实施计划），发现了 5 个关键的技术缺漏，已全部补充到相应的工作单中。

## 5 个关键技术缺漏及补充

### 1. 协议补全（最关键的缺漏）

**问题**：
- resolver 目前只实现了 TCP 和 Unix Socket
- 工作2 中虽然提到"实现 UDP 监听器启动"，但 resolver 内部的 Transport 接口定义目前是面向流（Stream）的
- 仅仅开启端口是不够的，报文的收发逻辑也需要适配 UDP

**补充位置**：工作2-递归模块启动集成

**具体补充**：
- 2.2 新增"重构 resolver 的 Transport 层以支持 UDP"
  - 分析 resolver/transport.go 的当前设计
  - 为 Transport 接口增加 UDP 支持
  - 实现 UDPTransport 结构体
  - 修改 resolver/server.go 的处理逻辑
  - 添加 UDP 监听器启动逻辑

---

### 2. 配置结构的"主权"回归

**问题**：
- resolver 目录下目前有一个独立的 config.go
- 工作1 提到创建 recursive.json，但没有明确处理 resolver 内部的配置
- 会出现两个包各玩各的配置，导致维护灾难

**补充位置**：工作1-配置项整合

**具体补充**：
- 1.2 新增"重构 resolver 包配置结构"
  - 移除 resolver/config.go
  - 修改 resolver.Resolver 结构体，移除内部配置字段
  - 修改 resolver.NewServer() 函数签名，接收主程序的配置结构体
  - 编写配置转换函数，将主程序的 RecursiveConfig 映射到 resolver 的配置结构

---

### 3. Root Hints 的加载机制

**问题**：
- 目前 getRootNameservers() 是写死的
- 工作1 提到了解析 named.cache，但没有明确如何注入到 resolver 中
- 在查询时动态读取磁盘文件会有性能问题

**补充位置**：工作1-配置项整合

**具体补充**：
- 1.5 新增"实现 named.cache 文件解析和 Root Hints 注入"
  - 为 resolver.Resolver 结构体增加 RootHints 字段
  - 在初始化时注入解析后的 Root Hints 对象（而不是在查询时动态读取磁盘）
  - 修改 resolver 中的 getRootNameservers() 函数，使用注入的 RootHints

---

### 4. 日志系统的"血缘"注入

**问题**：
- resolver 目前使用的是自己的（或全局默认的）logger
- 工作4 提到了分类日志，但没有明确如何将主程序的 Logger 注入到 resolver 中
- 无法通过主程序统一控制日志级别，也无法自动带上 [REC] 标签

**补充位置**：工作2-递归模块启动集成

**具体补充**：
- 2.3 新增"注入全局 Logger"
  - 在递归模块启动时注入主程序的 Logger 实例
  - 修改 resolver.NewServer() 函数，接收 Logger 参数
  - 修改 resolver 包中所有日志输出，使用注入的 Logger
  - 配置日志标签，确保所有日志都带有 [REC] 前缀

---

### 5. 迭代查询的并发优化（性能点）

**问题**：
- 当前迭代逻辑是顺序尝试 Nameservers
- 前 4 个工作项主要关注"能跑通"，未关注"跑得快"
- 递归解析的延迟主要在大批量迭代，并发探测能显著提升用户体验

**补充位置**：新增工作8.5-性能专项

**具体补充**：
- 新增工作8.5"性能专项 - 迭代查询并发优化"
  - 设计 Race 策略：对同一层级的多个 NS IP 并发发送查询
  - 实现 QueryNameserversRace 函数
  - 修改迭代逻辑集成并发
  - 实现智能 NS 排序（基于 RTT）
  - 实现连接复用
  - 添加并发配置和监控
