# 递归模块集成方案

## 概述

将独立的递归模块 `resolver` 集成到主 DNS 转发器中。递归功能作为一个等同于外部递归服务器的独立设计，通过本地 UDP 端口暴露，主转发器将其视为一个普通的上游服务器。

---

## 核心架构决策

### 1. 启动流程

**顺序**：配置初始化 → 启动递归线程 → 启动主转发器

**理由**：
- 转发器作为"客户端"，需要递归模块已就绪
- 如果启动瞬间递归模块还没起来，转发器会自动触发重试或切到备用 DNS
- 随主程序一起启动，但解耦初始化过程

**失败处理**：
- 如果递归线程启动失败，主程序拒绝启动
- 在启动日志中明确标记各阶段：`[INIT] Loading config...` → `[REC] Starting recursive resolver...` → `[FWD] Starting forwarder...`

---

### 2. 隐私原则

**严格执行递归，不泄露查询**

- 递归模块严格执行递归查询，不会向外部 DNS 泄露用户查询
- 如果递归失败，用户可通过配置其他上游服务器处理
- 用户在 Upstreams 中明确看到本地递归和其他服务器的优先级
- 这样明面体现会有 DNS 查询泄露减少带来的用户疑问

**失败场景**：
```
[REC] Query: example.com
[REC] Iterating... (trying root servers)
[REC] ERROR: All root hints failed (timeout/unreachable)
[REC] Query FAILED - No fallback to external DNS
[FWD] Return error to client or retry with configured timeout
```

用户选项：
- 配置更多的 Root Hints（国内镜像、自定义上游）
- 在 Upstreams 中添加其他 DNS 服务器作为备选
- 禁用本地递归，使用纯转发模式

---

### 3. 故障处理

**健康检查与自动降级**

- **故障感知**：主转发器已有对 Upstream 服务器的超时和健康评估机制。如果 127.0.0.1:5335 连续超时或返回 Connection Refused，转发器应将其标记为 Down，并自动切换到其他外部 Upstream
- **自愈机制**：递归模块作为独立线程，主线程应通过 channel 或 Status Flag 监控其生存状态。如果崩了，主线程可尝试重启该递归任务（Panic Recovery）
- **重启频率限制**：如果递归线程在 5 秒内崩溃超过 3 次，放弃重启，避免陷入重启循环

---

### 4. 配置管理

**独立配置文件，生命周期联动**

#### 配置文件结构

递归模块配置独立成单独的文件，实现真正的模块化：

```
config/
├── config.json              # 主程序配置（转发器相关）
├── recursive.json           # 递归模块配置（独立）
└── adblock.json            # 广告拦截配置（已有）
```

#### config.json 中的 Upstreams 配置

主程序配置中只保留上游服务器配置，递归模块的具体参数由 `recursive.json` 管理：

```json
{
  "upstreams": [
    {
      "address": "127.0.0.1:5335",
      "type": "local_recursive",
      "priority": 1
    },
    {
      "address": "8.8.8.8:53",
      "type": "external",
      "priority": 2
    },
    {
      "address": "1.1.1.1:53",
      "type": "external",
      "priority": 3
    }
  ]
}
```

#### 配置验证

**启动阶段的 ValidateConfig**：

1. 主程序读取 `config.json`，检查 Upstreams 中是否有 `type: local_recursive` 的项
2. 如果有，读取 `recursive.json` 检查 `enabled` 字段
3. 如果 Upstreams 包含本地递归端口但 `recursive.json` 中 `enabled` 为 false，程序拒绝启动并打印明确的错误：

```
[ERROR] Configuration Conflict: Local recursive resolver is specified in upstreams (127.0.0.1:5335) but recursive module is disabled in recursive.json. 
Please enable 'enabled' in recursive.json or remove the local_recursive upstream from config.json.
```

**配置验证流程**：
```
[INIT] Validating configuration...
[INIT] Loading config.json...
[INIT] Found upstream: 127.0.0.1:5335 (type: local_recursive)
[INIT] Loading recursive.json...
[INIT] Checking: recursive.enabled = true ✓
[INIT] Checking: recursive.port = 5335 ✓
[INIT] Checking: No port conflicts ✓
[INIT] Configuration validation passed
```

#### 配置独立性的优势

- **真正的独立性**：递归模块有自己的配置文件，不依赖主程序的配置结构
- **模块化**：递归模块可以独立演进，配置变更不影响主程序
- **清晰的职责边界**：主程序只需要知道"递归模块在哪个端口"，具体怎么配置是递归模块自己的事
- **易于维护**：用户可以单独调整递归模块的参数，而不需要修改主配置

### 5. 端口配置

**默认端口：5335**

- 避开 5353（mDNS 默认端口，容易冲突）
- 必须可配置
- 如果端口被占用，递归模块启动失败，应在 UI 上给出醒目标记，并提醒用户更换端口

### 6. Root Hints 策略

**使用官方 named.cache 文件作为根服务器源**

#### Root Hints 来源

- 使用官方 `named.cache` 文件，包含所有根服务器的完整信息（IPv4/IPv6）
- 源文件地址：https://www.internic.net/domain/named.cache
- 文件位置：`resolver/named.cache`（当前已存放位置）

#### 技术实现细节

- **解析逻辑**：利用项目中已收录的 `github.com/miekg/dns` 库提供的 Zone File 解析功能，直接加载并解析 `named.cache`。避免使用复杂的正则表达式，确保对 IPv4、IPv6 记录解析的健壮性。
- **传输层复用**：
    - 递归模块将复用项目已有的 **DoH (DNS over HTTPS)** 和 **DoT (DNS over TLS)** 查询代码逻辑。
    - 虽然根服务器主要支持 UDP/TCP，但在迭代查询过程中，如果对应的权威服务器支持加密协议，可以利用已有的传输层能力提升穿透力。
    - 为确保不被 UDP 拦截干扰，递归模块可以灵活选择 TCP 或加密协议作为传输兜底，但核心迭代路径严格遵循 `named.cache` 所指示的根和权威节点。

#### 根服务器更新机制

**初期**：手动将 `named.cache` 文件放入 `resolver/` 目录。

**后期**：在 Web 界面中添加"更新根目录"功能
- 显示当前 `named.cache` 的更新时间
- 提供"更新根目录"按钮，从官方地址下载最新版本
- 更新后自动重新加载

#### 防污染策略

- 仅使用官方根服务器作为迭代起点，从根结点开始建立信任链。
- 结合 DoH/DoT 传输能力的复用，最大限度穿透 UDP 拦截，确保递归查询的真实性与隐私性。

---

### 7. 缓存策略

**双重缓存设计**

- **递归模块缓存**：用于加速迭代过程（TTL 可较短，如 5 分钟）
- **Forwarder 缓存**：用于加速最终结果（TTL 遵循 DNS 记录的 TTL）

**性能考量**：
- 虽然有两次查询，但因为是 Loopback 且本地内存操作，性能损失极小
- 两个模块的逻辑最干净，互不干扰

### 9. 模块感知设计

**方案 B：感知增强**

主程序知道 127.0.0.1:5335 是"自己人"。

#### 实现方式

在 Upstream 配置中添加 `type` 字段：

```json
{
  "address": "127.0.0.1:5335",
  "type": "local_recursive",
  "priority": 1
}
```

#### 感知的好处

- **用户可见性**：Dashboard 上明确显示"Local Recursive Resolver"的状态，而不是一个模糊的 IP 地址
- **故障诊断**：用户能立即知道是本地递归挂了，而不是去怀疑网络或外部 DNS
- **监控和告警**：可针对"本地递归"设置专门的告警规则
- **配置验证**：启动时可明确检查"本地递归是否成功启动"
- **代码可维护性**：主程序知道 5335 的特殊性，可在代码中明确标注这是"内部服务"

#### 启动时的感知

```
[INIT] Validating configuration...
[INIT] Found upstream: 127.0.0.1:5335 (type: local_recursive)
[INIT] Checking: recursive_enabled = true ✓
[INIT] Checking: recursive module will start on port 5335 ✓
[INIT] Checking: No port conflicts ✓
[INIT] Configuration validation passed
```

### 设计理念

既然递归模块是作为一个"独立外部服务器"设计的，前端也采用"既整合又独立"的思路，通过专业的可视化界面提升用户体验和"专业感"。

**独立管理选项卡**：
- 不要只是把递归配置放在"设置"里的一个长表单中
- 给递归模块一个独立的管理选项卡

**状态实时监控**：
```

├─ 运行状态：✅ 运行中 (Uptime: 2h 34m)

│     ├─ 3层迭代：45%

├─ 最近错误：无
└─ 操作
   ├─ [重启] [停止] [日志]

**交互式迭代路径跟踪**：
- 提供一个调试框，用户输入域名（如 example.com）
- 前端展示完整的递归路径：
  ```

- 这能让用户直观感受到递归的"隐私性"和工作原理

- 显示当前 named.cache 的版本信息
- 显示根服务器节点数量（通常 13 个）
- 显示文件的最后修改时间和下载来源

- 对 13 个根服务器分别显示"连通性检查"
- 支持"快速诊断"按钮，一键检测所有根服务器的连通性
- 如果用户发现大部分根服务器变红，能快速判断是由于根服务器被 UDP 阻断导致的解析变慢

- 带进度条的一键更新

- 更新后自动重新加载

### 6. 递归模块的"教育模式"

提供一个"学习模式"，用户可以：
- 输入域名
- 选择"显示完整迭代过程"
- 看到每一步的 DNS 查询和响应
- 理解递归的工作原理

这能大幅提升用户对隐私保护的理解和信任。

**示例**：
```
学习模式 - 查询 example.com

  发送：example.com? → a.root-servers.net
  响应：请查询 .com 服务器 (192.5.6.30)
  耗时：45ms

第 2 步：查询 TLD 服务器
  发送：example.com? → 192.5.6.30 (.com 服务器)
  响应：请查询 example.com 服务器 (93.184.216.1)
  耗时：38ms

第 3 步：查询权威服务器
  发送：example.com? → 93.184.216.1
  响应：example.com A 93.184.216.34
  耗时：32ms

总耗时：115ms
缓存状态：新查询（无缓存）
隐私状态：✅ 完全隐私（查询未离开本地）
```
